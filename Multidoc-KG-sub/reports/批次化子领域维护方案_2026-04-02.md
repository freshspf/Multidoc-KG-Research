# 批次化子领域维护方案（2026-04-02）

本文档用于解决当前子领域体系在“大批量运行”场景下的核心问题：

1. 如何避免每来一篇论文就即时改动整棵子领域层级树
2. 如何保证每一批论文的分类依据稳定、可追踪、可复盘
3. 如何把“候选新子领域”与“正式子领域”区分开来
4. 如何在 Neo4j 中维护可版本化的 biomedical taxonomy

---

## 1. 问题背景

当前系统已经具备以下能力：

- 每篇论文可以分配一个 `subdomain`
- `Paper / Subdomain / CLASSIFIED_AS / SUBCLASS_OF` 已经可以写入 Neo4j
- 分类时会优先复用已有子领域，避免节点爆炸

但当前模式仍然是“在线生长”：

```text
来一篇论文
-> 子领域分类
-> 必要时新建 Subdomain
-> 立即写入 hierarchy
```

这个模式在小规模验证阶段可以接受，但在大批量运行下存在明显风险：

- 不同批次会因为已有 hierarchy 不同而得到不同分类结果
- 新节点会在不同时间点以不同命名风格进入图谱
- 批次顺序会影响 taxonomy 结构
- 很难回答“这一批论文是基于哪一版子领域体系分类的”
- 很难区分“正式子领域”与“临时候选标签”

---

## 2. 总体原则

后续不再采用“每篇论文即时重构 hierarchy”的方式，而改为：

```text
批次论文
-> 使用当前 taxonomy 版本做分类
-> 大部分论文复用已有 confirmed subdomains
-> 少量论文产生 candidate subdomains
-> 批次结束后统一做 hierarchy refinement
-> 发布新的 taxonomy version
```

一句话概括：

> 一批一批地分类，一批一批地整理子领域层级。

---

## 3. 新的运行模式

### 3.1 分类阶段

每一批论文进入系统时，不直接自由修改正式 hierarchy，而是：

- 优先复用已有正式子领域
- 如果无法复用，则生成 `candidate_subdomain`
- candidate 先进入候选区，不立即成为正式 taxonomy 的一部分

分类阶段的目标不是“重构整棵树”，而是回答两个问题：

1. 这篇论文是否可以归入现有 confirmed subdomain
2. 如果不能，应该生成什么 candidate subdomain

### 3.2 批次结束后的整理阶段

当一批论文处理完后，再单独运行 `hierarchy refinement`：

- 检查本批新增的 candidate subdomains
- 与已有 confirmed subdomains 做近义归并
- 决定 candidate 是否升级为 confirmed
- 决定其 `SUBCLASS_OF` 应该挂到哪个父节点
- 更新 taxonomy 版本号

这样分类和层级演化被拆成了两步：

- 论文级判断
- taxonomy 级重组

---

## 4. 推荐的数据状态划分

建议把子领域对象分成 3 类状态：

### 4.1 Confirmed Subdomain

正式子领域，允许后续批次直接复用。

特点：

- 已进入正式 hierarchy
- 可作为 `CLASSIFIED_AS` 的目标节点
- 可作为其他 subdomain 的父节点
- 属于某个明确的 taxonomy 版本

### 4.2 Candidate Subdomain

候选子领域，仅表示某一批分类时发现了可能需要的新节点。

特点：

- 还未正式进入 confirmed hierarchy
- 不应直接影响后续所有批次的分类结果
- 可以带有来源论文、置信度、建议父节点、建议别名等信息

### 4.3 Taxonomy Version

用于标识某篇论文的分类，是基于哪一版子领域体系得出的。

特点：

- 每次 hierarchy refinement 完成后递增
- 论文分类结果应关联到当时使用的 taxonomy version
- 便于复盘、对比不同版本的分类差异

## 5. 推荐的 Neo4j 图模型扩展

### 5.1 正式子领域节点

```text
(:Subdomain {
  name,
  normalized_name,
  status: "confirmed",
  taxonomy_version,
  created_at,
  updated_at
})
```

### 5.2 候选子领域节点

```text
(:SubdomainCandidate {
  name,
  normalized_name,
  suggested_parent,
  confidence,
  source_batch_id,
  source_taxonomy_version,
  status: "candidate",
  created_at,
  updated_at
})
```

### 5.3 论文节点新增字段

```text
(:Paper {
  id,
  title,
  subdomain,
  parent_domain,
  taxonomy_version,
  batch_id,
  subdomain_confidence,
  subdomain_reason
})
```

### 5.4 推荐关系

正式关系：

- `(Paper)-[:CLASSIFIED_AS]->(Subdomain)`
- `(Subdomain)-[:SUBCLASS_OF]->(Subdomain)`

候选关系：

- `(Paper)-[:SUGGESTS_SUBDOMAIN]->(SubdomainCandidate)`
- `(SubdomainCandidate)-[:CANDIDATE_SUBCLASS_OF]->(Subdomain)`
- `(SubdomainCandidate)-[:MERGED_INTO]->(Subdomain)`（如果后续被归并）

版本关系：

- `(TaxonomyVersion)-[:INCLUDES]->(Subdomain)`
- `(Paper)-[:CLASSIFIED_UNDER]->(TaxonomyVersion)`

如果不想第一版就引入 `TaxonomyVersion` 节点，也至少要保留：

- `Paper.taxonomy_version`
- `Subdomain.taxonomy_version`
- `SubdomainCandidate.source_taxonomy_version`

---

## 6. 推荐的批次运行流程

### 6.1 批次开始前

- 读取当前 confirmed hierarchy
- 读取当前 taxonomy version
- 锁定本批次使用的 taxonomy version

### 6.2 本批论文分类

对每篇论文：

1. 优先匹配已有 confirmed subdomain
2. 如果能匹配：
   - 写 `Paper.subdomain`
   - 建立 `CLASSIFIED_AS`
   - 记录 `taxonomy_version`
3. 如果不能匹配：
   - 生成 `candidate_subdomain`
   - 建立 `SUGGESTS_SUBDOMAIN`
   - 暂不改正式 hierarchy

### 6.3 批次结束后

运行 `hierarchy refinement`：

1. 汇总本批 candidate subdomains
2. 合并近义 candidate
3. 判断哪些 candidate 需要升级为 confirmed
4. 为升级节点选择合适父节点
5. 更新正式 `SUBCLASS_OF`
6. 发布新 `taxonomy_version`

### 6.4 下一批开始时

下一批论文使用新的 confirmed hierarchy 和新的 taxonomy version。

---

## 7. 为什么这种方案更适合大批量运行

### 7.1 分类依据稳定

每一批论文都基于一个固定的 taxonomy version 分类，而不是边跑边变。

### 7.2 新节点受控

不会因为某一篇论文的偶然表达就立即污染正式 hierarchy。

### 7.3 易于人工检查

人工只需要检查：

- 本批 candidate 列表
- candidate 到 confirmed 的归并结果
- 新增的 `SUBCLASS_OF`

而不是逐篇追踪所有 hierarchy 改动。

### 7.4 易于汇报与复盘

可以清楚回答：

- 第几批论文主要分到了哪些 confirmed subdomains
- 本批新增了哪些 candidate subdomains
- 哪些 candidate 被并回已有领域
- 哪些真正升级成了新的正式子领域

## 8. 推荐的实现顺序

### 第一阶段：最小批次控制

在现有系统上先补：

- `Paper.taxonomy_version`
- `Subdomain.status`
- `SubdomainCandidate` 节点
- `SUGGESTS_SUBDOMAIN` 关系

这一步的目标是：

- 区分 confirmed 和 candidate
- 不再让所有新 subdomain 直接进入正式 hierarchy

### 第二阶段：批次级 refinement

新增一个 `SubdomainHierarchyRefinementAgent`，专门处理：

- candidate 聚合
- candidate 与 confirmed 的归并
- 父节点选择
- 新 taxonomy version 发布

### 第三阶段：版本化运行

在主流程中加入：

- `batch_id`
- `taxonomy_version`
- 按批次运行分类
- 按批次运行 refinement

最终形成：

```text
Batch N
-> classify papers under taxonomy vK
-> store candidate subdomains
-> refine hierarchy
-> publish taxonomy vK+1
```

---

## 9. 与当前实现的关系

当前实现已经完成的部分：

- `Paper / Subdomain / CLASSIFIED_AS / SUBCLASS_OF` 基础落库
- hierarchy-aware 子领域分类
- 近义标签复用
- 基础名称规范化

当前尚未完成的关键部分：

- confirmed/candidate 分层
- taxonomy version
- 批次级 hierarchy refinement
- candidate 升级与归并机制

因此，后续代码改造应当优先处理：

1. 子领域状态分层
2. taxonomy version 持久化
3. hierarchy refinement agent

而不是继续强化“单篇论文即时生长 hierarchy”的逻辑。

---

## 10. 当前建议结论

后续大批量运行时，推荐采用以下工作模式：

```text
论文批次
-> 使用固定 taxonomy version 进行分类
-> 复用 confirmed subdomains
-> 记录 candidate subdomains
-> 批次结束后统一 refinement
-> 发布下一版 taxonomy
```

这意味着：

- 子领域体系不再实时无约束生长
- taxonomy 会像本体/分类体系一样按版本演化
- 每一批论文的分类结果都能被清楚解释和追踪

这将比当前“在线自由生长”的方式更适合正式实验和后续扩展。
