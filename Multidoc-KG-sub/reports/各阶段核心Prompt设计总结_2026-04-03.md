# 各阶段核心 Prompt 设计总结

## 1. 文档目的

本文档总结当前医学文献知识图谱主线中，各个 LLM 驱动阶段的核心 prompt 设计思路，便于：

- 阶段性汇报
- 后续 prompt 调整
- 其他 AI 或研究者快速恢复上下文

当前主线中涉及 prompt 的主要阶段包括：

- 子领域识别
- 子领域 refinement
- 三元组抽取
- 实体接地
- 知识验证

`Evolution / 入库` 阶段主要是规则化写库，不是 prompt 驱动阶段。

---

## 2. 子领域识别（Subdomain Classification）

对应代码：

- `/Users/joer/Gitroom/Multidoc-KG-zya/agents/subdomain.py`

### 2.1 阶段目标

给每篇医学论文分配一个尽量合适的 biomedical subdomain，并判断该 subdomain 是否已经存在于当前 hierarchy 中。

### 2.2 Prompt 角色设定

系统角色被设定为：

- `experienced ontology engineering expert`

也就是说，该 prompt 不是把模型当成普通分类器，而是当成一个“能够在现有层级结构中做概念定位和扩展”的本体工程专家。

### 2.3 主要输入

该阶段 prompt 的输入主要包括三部分：

1. 顶层领域
- `Top Domain = Biomedicine`

2. 当前已有层级
- `[Given Hierarchy]`
- 由 Neo4j 中现有 `Subdomain -[:SUBCLASS_OF]-> Subdomain` 关系动态读取

3. 论文文本
- `[Given Text]`
- 主要由 `title + abstract + keywords` 组成
- 如果标题和摘要信息不足，则补充内容摘要

### 2.4 核心任务

模型需要完成四件事：

1. 找到最能反映文本内容的 biomedical subdomain
2. 判断该 subdomain 是否已经存在于当前 hierarchy 中
3. 如果已存在，则不新增关系
4. 如果不存在，则给出其最合适的父概念，并输出需要补入的 `subclass_of` 关系

### 2.5 关键约束

当前 prompt 明确强调：

- 优先使用 `title / abstract / keywords` 作为判断信号
- 优先参考已有 hierarchy，而不是固定候选列表
- subdomain 不能过宽，也不能过度 paper-specific
- 如果已有 hierarchy 中存在语义非常接近的概念，应优先复用
- 只有在没有合适已有概念时，才允许提出新 subdomain
- `parent_domain` 必须是最接近的更宽概念
- 关系标签严格使用 `subclass_of`

### 2.6 输出格式

输出为严格 JSON，核心字段包括：

- `subdomain`
- `parent_domain`
- `reason`
- `confidence`
- `new_relations`

### 2.7 设计特点

这一阶段的 prompt 本质上不是传统“文本分类”，而是：

- `hierarchy-aware classification`
- `taxonomy extension suggestion`

也就是说，它既负责“分到哪里”，也负责“如果没有这个概念，应该怎么挂进现有层级”。

## 3. 子领域 Refinement

对应代码：

- `/Users/joer/Gitroom/Multidoc-KG-zya/agents/subdomain_refinement.py`

### 3.1 阶段目标

对分类阶段产生的 `SubdomainCandidate` 做进一步整理，决定：

- 该 candidate 应并入已有 confirmed subdomain
- 还是应升级为新的 confirmed subdomain

### 3.2 Prompt 角色设定

系统角色同样是：

- `ontology engineering expert`

但这一阶段不是处理论文文本，而是处理已经产生的一批候选概念。

### 3.3 主要输入

该阶段的 prompt 输入包括：

1. 已确认的正式子领域
- `[Confirmed Subdomains]`

2. 待处理候选子领域
- `[Candidate Subdomains]`

每个 candidate 一般带有这些信息：

- 候选名称
- 建议父节点
- 命中的论文数量
- 平均置信度
- 样例论文

### 3.4 核心任务

模型要对每个 candidate 做出二选一决策：

1. `merge`
- 并入某个已有 confirmed subdomain

2. `promote`
- 升级为一个新的 confirmed subdomain

### 3.5 关键约束

当前 refinement prompt 明确强调：

- 如果 candidate 与已有 confirmed 概念语义等价，优先 `merge`
- 如果 candidate 更像研究主题、方法词、流程词、实践词，也优先 `merge`
- 只有当 candidate 是稳定、可复用、医学语义明确的子领域时才 `promote`
- 疾病名、肿瘤类型、临床稳定专科，更倾向于 `promote`
- `merge` 的目标必须是已有 confirmed subdomain
- `promote` 时必须给出最合适的 confirmed parent

### 3.6 输出格式

输出为严格 JSON，核心字段包括：

- `candidate`
- `action`
- `target_subdomain`
- `parent_domain`
- `reason`

### 3.7 设计特点

这一阶段不是论文理解 prompt，而是：

- `candidate lifecycle management`
- `taxonomy cleanup / consolidation`

它的作用是避免子领域体系在大批量运行时无限制爆炸。

## 4. 三元组抽取（Extraction）

对应代码：

- `/Users/joer/Gitroom/Multidoc-KG-zya/agents/extraction.py`

### 4.1 阶段目标

从医学论文文本中抽取知识三元组，并区分：

- ontology layer
- instance layer

### 4.2 Prompt 角色设定

系统角色设定为：

- `biomedical literature annotation expert`

### 4.3 主要输入

当前 extraction prompt 的输入包括：

1. 论文上下文 `Paper context`
- title
- assigned subdomain
- parent domain
- abstract
- keywords

2. biomedical few-shot examples

3. 当前 chunk 文本

### 4.4 核心任务

模型要同时抽取两层知识：

1. `Ontology layer`
- 概念层、可复用层知识
- 如疾病分类、药物类型、biomarker 关系

2. `Instance layer`
- 论文/病例/实验/队列层具体事实
- 如患者诊断、治疗行为、实验观察结果

### 4.5 当前 prompt 的关系约束

这一阶段已经不再允许自由关系生成，而是使用受控关系集合。

Ontology 允许的关系主要包括：

- `subclass_of`
- `type`
- `biomarker_for`
- `risk_factor_for`
- `treats`
- `measured_in`
- `part_of`
- `associated_with`
- `inhibits`
- `activates`
- `causes`
- `predicts`

Instance 允许的关系主要包括：

- `has_diagnosis`
- `received_treatment`
- `combined_with`
- `improves`
- `reduces`
- `increases`
- `associated_with`
- `predicts`
- `has_adverse_event`
- `measured_value`
- `measured_in`

### 4.6 关键约束

当前 prompt 明确要求：

- 只能抽取文本中明确支持的事实
- `subdomain` 只能作为 soft prior，不允许借此脑补事实
- 实体名称尽量贴近原文，不做额外归一化
- 不允许把 `study / analysis / we / findings / figure / table` 这类内容作为实体
- 不允许抽取作者信息、基金、图表标签、页眉页脚、参考文献
- 必须返回严格 JSON

### 4.7 输出格式

输出严格分为两部分：

- `ontology`
- `instances`

每条 claim 统一包含：

- `subject`
- `relation`
- `object`
- `evidence`

### 4.8 设计特点

这一阶段的 prompt 设计核心在于：

- 从“开放式信息提取”改为“受控 schema 提取”
- 同时显式利用 `subdomain` 提供语境约束

也就是说，子领域不会直接决定抽什么，但会影响模型如何理解术语和文本语境。

## 5. 实体接地（Grounding）

对应代码：

- `/Users/joer/Gitroom/Multidoc-KG-zya/agents/grounding.py`

### 5.1 阶段目标

将 extraction 阶段得到的实体与图中已有节点对齐，判断：

- 是否与已有节点合并
- 或需要创建新节点

### 5.2 该阶段不是纯 prompt 驱动

这一阶段采用的是：

- 向量检索候选
- LLM 做最终判断

因此 prompt 只承担“最后一跳决策”，而不是从零开始做 grounding。

### 5.3 主要输入

grounding prompt 的输入包括：

1. 目标实体
- `target entity`

2. 候选实体列表
- 带 `id`
- 带相似度得分

3. 上下文信息
- evidence
- paper title
- paper subdomain
- section title
- claim_type

### 5.4 概念与实例两套 prompt

当前 grounding prompt 分两类：

1. 概念接地
- 判断目标概念是否应与已有概念合并

2. 实例接地
- 判断目标实例是否应与已有实例合并

### 5.5 核心规则

概念接地强调：

- 只有真正等义/别名才合并
- 上下位概念不能直接合并
- 医学术语粒度必须一致

实例接地强调：

- 只有指向同一个具体对象时才合并
- 不同患者、不同队列、不同样本不能因为名字相近就强行合并
- `patient / cohort` 这类泛称默认更谨慎

### 5.6 输出格式

输出为 JSON，核心是：

- `decision = merge | new`
- 若为 merge，则带 `id`
- 再给简短 `reasoning`

### 5.7 设计特点

这一阶段的 prompt 本质是：

- `candidate-constrained entity resolution`

也就是模型不是自由决定实体是什么，而是在有限候选中做“合并还是新建”的判定。

## 6. 知识验证（Validation）

对应代码：

- `/Users/joer/Gitroom/Multidoc-KG-zya/agents/validation.py`

### 6.1 阶段目标

判断新提取并接地后的 claim 是否应被接受进入图谱。

### 6.2 Prompt 角色设定

系统角色设定为：

- `医学文献知识图谱验证专家`

### 6.3 主要输入

当前 validation prompt 会使用：

1. 新 claim 本身
- `subject relation object`

2. 证据原文

3. 论文上下文
- title
- subdomain
- parent_domain
- section

4. 图谱已有相关知识
- historical context

### 6.4 核心任务

模型需要判断该 claim：

- 是否有效
- 与已有知识是 `support`
- 还是 `conflict`
- 或属于 `new`

如果证据不足、关系不清、实体异常，则直接拒绝。

### 6.5 关键约束

当前 validation prompt 强调：

- 优先依据证据原文
- 不要补全证据外信息
- ontology claim 要关注概念层次和方向是否合理
- instance claim 要关注病例/实验/观察是否被证据支持
- 若与历史知识明显一致则 `support`
- 若明显冲突则 `conflict`
- 若证据支持但图中暂无相似知识则 `new`

### 6.6 输出格式

单条验证输出：

- `valid`
- `type`
- `confidence`
- `reasoning`

批量验证时返回 JSON 数组，每项对应一条 claim。

### 6.7 设计特点

这一阶段是：

- `evidence-first acceptance / rejection`

也就是说，它的核心不是给 claim 重新抽取含义，而是判断：

- 这条 claim 是否被证据支持
- 是否与已有图谱发生冲突
- 是否值得进入最终图谱

## 7. Evolution / 入库

对应代码：

- `/Users/joer/Gitroom/Multidoc-KG-zya/agents/evolution.py`

### 7.1 阶段特点

该阶段基本不是 prompt 驱动，而是规则化写库阶段。

主要职责是：

- 接收通过 validation 的 claim
- 调用 Neo4j 写库逻辑
- 按 ontology / instance 类型落盘

因此在汇报时，可以将其描述为：

- `规则化图谱演化阶段`

而不是 prompt 设计阶段。

---

## 8. 总结

当前整条主线的 prompt 设计逻辑可以概括为：

1. 子领域识别
- 用 hierarchy-aware prompt 先给论文分配领域语境

2. 子领域 refinement
- 将 candidate 整理成 confirmed taxonomy

3. 三元组抽取
- 在受控 biomedical relations 下抽取 ontology / instance 两层知识

4. 实体接地
- 在候选实体集合上判断 merge 或 new

5. 知识验证
- 基于证据和历史图谱决定 claim 是否可接受

6. 演化入库
- 将通过验证的结果写入 Neo4j

从方法论上看，当前系统不是把一个 prompt 做到底，而是把任务拆成：

- 领域定位
- taxonomy 维护
- 受控抽取
- 候选约束接地
- 证据驱动验证
- 规则化入库

这是当前这套医学文献知识图谱主线最核心的 prompt 设计思路。
