# 子领域融合与关系维护改造方案（2026-04-02）

本文档用于回答当前项目中与“子领域”相关的三个核心问题：

1. 如何把子领域真正融合进现有系统，而不是只停留在 `paper.metadata`
2. 如何在 Neo4j 中维护子领域字段与子领域层级关系
3. 如何把当前“半静态配置 + 开放关系抽取”的做法，改造成更符合导师要求的方案

---

## 1. 导师要求的核心含义

根据当前理解，导师的原话可以拆成 3 个明确要求：

### 1.1 在 Neo4j 里加上子领域字段

这不应只理解为：

- 在 CSV 里多一列
- 在 `paper.metadata` 里多一个字段

更合理的理解应该是：

- `Paper` 节点要能查询到自己的 `subdomain / parent_domain`
- 图数据库里要能直接基于子领域检索论文、统计论文、聚合同类知识

### 1.2 维护子领域之间关系

这意味着子领域不是孤立标签，而应该形成层级，例如：

```text
Biomedicine
  -> Oncology
    -> Liver cancer immunotherapy
    -> Cancer liquid biopsy
    -> Cancer drug delivery systems
```

也就是说：

- 不只是给论文打标签
- 还要把子领域本身当成图中的对象来维护

### 1.3 子领域配置不能写死，要让大模型自己判断

这意味着当前这种做法是不够的：

- 固定一组 `parent_domains`
- 固定一组 `keyword_rules`
- 让模型只在这套候选中选

更接近导师要求的做法应当是：

- 让 LLM 读取“当前已有的领域层级”
- 判断论文是否应归到已有子领域
- 若不适合已有层级，再提出新的子领域与新的 `subClassOf` 关系

---

## 2. 当前实现现状

### 2.1 当前子领域模块怎么工作

当前链路为：

```text
PaperDataLoader
-> SubdomainClassifierAgent
-> 结果写回 paper.metadata
-> Extraction 读取 subdomain 作为 soft prior
```

当前代码位置：

- `agents/subdomain.py`
- `main.py`
- `schema.py`

当前输出字段：

- `subdomain`
- `parent_domain`
- `reason`
- `confidence`
- `new_relations`

但这些结果目前只：

- 写在运行时的 `paper.metadata`
- 导出到 `reports/subdomain_assignments.csv`

还没有：

- 持久化到 Neo4j
- 成为图中的正式节点和关系

### 2.2 当前子领域层级是怎么“维护”的

当前其实并没有真正维护层级。

目前的做法是：

- 模型输出 `subdomain`
- 模型输出 `parent_domain`
- 然后 `_coerce_relations()` 自动补一条：

```text
subdomain -[subclass_of]-> parent_domain
```

所以当前“层级关系”本质上是：

- 运行时补边
- 不是真正基于图谱已有 hierarchy 推理得出
- 也没有写入图数据库

### 2.3 当前配置为什么还算“写死”

当前 `config/subdomain_config.yaml` 中仍包含：

- 固定 `parent_domains`
- 固定 `keyword_rules`

作用上相当于：

- 先给模型一个静态候选集
- 再让模型在这个候选集附近判断

这与导师要求的“不要写死，要让大模型自己判断”不完全一致。

## 3. 当前关系抽取链路是怎么做的

导师还提到“仔细看一下他是怎么抽关系的”，这部分必须单独说清。

### 3.1 当前三元组关系是怎么抽出来的

当前在 `agents/extraction.py` 中：

- 模型被要求输出：
  - ontology claims
  - instance claims
- 每条 claim 结构为：

```json
{
  "subject": "...",
  "relation": "...",
  "object": "...",
  "evidence": "..."
}
```

关系词来源于两部分：

1. prompt 中给出的推荐关系词
2. 模型自由生成

之后 `_normalize_relation()` 会做少量归一化。

### 3.2 当前关系抽取的现实问题

虽然 prompt 已经要求“controlled relation labels”，但当前实验结果说明：

- 关系词仍然较开放
- 出现了很多自由关系，如：
  - `candidate_susceptibility_gene_for`
  - `may_contribute_to`
  - `analyzed_in_study`
  - `demonstrated`
  - `used_for`
  - `used_in`

这意味着：

- 当前的“关系抽取”不是强 schema 驱动
- 更像“LLM 根据语义自然命名关系”

### 3.3 当前落库时怎么判断关系类型

在 `core/neo4j_store.py` 中，写库逻辑会调用：

- `_determine_claim_type()`

用来判断当前 claim 是：

- `ontology_subclass`
- `ontology_type`
- `ontology_property`
- `instance`

但这里当前仍然主要按旧规则判断，尤其偏向：

- 中文关系词
- 旧域遗留的 ontology 关系分类方式

这会带来一个实际问题：

- `Extraction` 现在输出英文 biomedical relations
- 但 `Neo4j` 侧对 relation type 的理解与之并未完全对齐

也就是说：

> 当前三元组“抽关系”和“落关系”还没有完全统一。

---

## 4. 当前系统与导师要求之间的主要缺口

### 缺口 1：子领域还不是图中的正式对象

现在子领域只是：

- 运行时标签
- CSV 导出字段

不是：

- 图中的 `Subdomain` 节点
- 也不是稳定层级中的一部分

### 缺口 2：层级维护仍然是静态候选 + 默认补边

这不是真正的“维护 hierarchy”，只是：

- 模型给一个标签
- 程序补一条 `subclass_of`

### 缺口 3：没有从 Neo4j 反向读取当前 hierarchy 参与分类

导师要求“大模型自己判断”，更合理的理解是：

- 模型应该在“已有图谱层级”的上下文中判断
- 而不是只在配置文件中给定候选

### 缺口 4：关系抽取与关系落库没有统一 schema

当前 `Extraction -> Neo4jStore` 之间存在错位风险：

- 前者偏开放
- 后者偏规则分类

---

## 5. 推荐的总体改造方向

建议把子领域从“辅助标签”升级为“图中的正式层级对象”，并形成下面这条新链路：

```text
论文输入
-> 子领域识别
-> 写入 Paper 节点 + Subdomain 节点 + Subdomain 层级
-> Extraction（读取子领域上下文）
-> Grounding
-> Validation
-> Evolution（写 claim）
```

这里有一个关键变化：

- 子领域信息不再只是给 Extraction 临时用
- 而是正式成为图谱的元结构信息

---

## 6. 推荐的 Neo4j 图模型

### 6.1 节点设计

建议新增并显式维护以下两类节点：

#### Paper 节点

```text
(:Paper {
  id,
  title,
  doi,
  journal,
  year,
  subdomain,
  parent_domain,
  subdomain_confidence,
  subdomain_reason,
  created_at,
  updated_at
})
```

作用：

- 让论文本身成为可查询对象
- 支持直接按 `subdomain` 聚合论文

#### Subdomain 节点

```text
(:Subdomain {
  name,
  normalized_name,
  level,
  source,
  created_at,
  updated_at
})
```

作用：

- 将子领域作为图谱中的正式概念层级来维护

### 6.2 关系设计

建议新增以下关系：

#### Paper 到 Subdomain

```text
(p:Paper)-[:CLASSIFIED_AS {
  confidence,
  reason,
  source: "llm"
}]->(s:Subdomain)
```

#### Subdomain 层级关系

```text
(child:Subdomain)-[:SUBCLASS_OF {
  source: "llm" | "manual"
}]->(parent:Subdomain)
```

可选：

```text
(:Subdomain {name: "biomedicine"})
```

作为整棵层级树的根。

### 6.3 为什么要同时保留字段和关系

建议 **Paper 节点字段** 和 **Subdomain 节点关系** 同时保留。

原因：

- 字段适合快速查询和导出
- 关系适合维护层级、遍历 hierarchy、做领域聚合

所以：

- `Paper.subdomain` 保留
- `Paper -[:CLASSIFIED_AS]-> Subdomain` 也保留

这不是重复，而是两种查询层次。

## 7. 推荐的子领域维护机制

这是整个方案中最关键的部分。

### 7.1 目标

让子领域分类从“静态候选选择”升级为“动态图谱驱动分类”。

### 7.2 推荐流程

#### Step 1：从 Neo4j 读取当前 hierarchy

分类前，读取当前已有的：

- `Subdomain` 节点
- `SUBCLASS_OF` 关系

然后整理成文本 hierarchy，例如：

```text
Biomedicine
  Oncology
    Liver cancer immunotherapy
    Cancer liquid biopsy
  Pathology and molecular diagnostics
    Single-cell RNA sequencing computational methods
```

#### Step 2：把 hierarchy 提供给 LLM

给 LLM 的输入应包括：

- 论文 `title + abstract + keywords`
- 必要时的 `content summary fallback`
- 当前已有 hierarchy

然后让模型判断：

1. 该论文属于哪个已有子领域？
2. 如果没有合适已有子领域，应该提出哪个新子领域？
3. 这个新子领域应挂在哪个父领域下面？

#### Step 3：返回结构化结果

建议返回：

```json
{
  "paper_subdomain": "...",
  "parent_domain": "...",
  "is_new_subdomain": true,
  "reason": "...",
  "confidence": 0.92,
  "new_nodes": [
    {"name": "cancer liquid biopsy"}
  ],
  "new_relations": [
    {"subject": "cancer liquid biopsy", "relation": "subclass_of", "object": "oncology"}
  ]
}
```

#### Step 4：写回图谱

分类完成后立即执行：

- `MERGE Paper`
- `MERGE Subdomain`
- `MERGE SUBCLASS_OF`
- `MERGE CLASSIFIED_AS`

这样 hierarchy 会随着论文持续扩展。

## 8. 配置如何改成“不写死”

### 8.1 当前不合理点

当前 `config/subdomain_config.yaml` 里存在：

- 大量固定 `parent_domains`
- 大量固定 `keyword_rules`

这意味着系统更像：

- “规则辅助分类器”

而不是：

- “基于图谱现状的 LLM 分类器”

### 8.2 推荐改法

把配置文件从“领域词表”改成“运行参数”。

建议保留的配置只有：

```yaml
max_input_chars: 6000
max_hierarchy_depth: 4
max_existing_subdomains: 200
allow_new_subdomain: true
root_domain: biomedicine
fallback_parent_domain: biomedicine
```

也就是说：

- 不再在配置里写死所有父领域
- 不再维护大量关键词规则
- hierarchy 的主要来源改为 Neo4j

### 8.3 是否完全不要 fallback

不建议完全删除 fallback，但 fallback 应该降级：

- 仅在 LLM 失败时使用
- 只提供兜底，不作为主分类依据

可以保留一个极简 fallback：

- oncology
- infectious diseases
- pathology and molecular diagnostics
- public health and epidemiology

但不要再维护大规模静态词表。

## 9. 如何融合进现有系统

### 9.1 推荐改造位置

#### `main.py`

当前：

- 子领域分类完成后，只写回 `paper.metadata`

建议改成：

- 子领域分类完成后，立即写入 `Paper + Subdomain + hierarchy`

#### `agents/subdomain.py`

当前：

- 从静态 YAML 读父领域与关键词规则

建议改成：

- 从 Neo4j 读取当前 hierarchy
- 将 hierarchy 作为 prompt 输入
- 输出“归属已有子领域”或“新增子领域”

#### `core/neo4j_store.py`

新增方法：

- `write_paper_node(paper, assignment)`
- `write_subdomain_assignment(paper_id, assignment)`
- `write_subdomain_hierarchy(relations)`
- `get_subdomain_hierarchy()`

### 9.2 是否需要新 agent

有两种选择：

#### 方案 A：继续放在 main.py 里直接调用

优点：

- 改动最小

缺点：

- 子领域写图逻辑会堆在主入口

#### 方案 B：新增 `SubdomainEvolutionAgent`

职责：

- 专门负责把子领域分类结果写入 Neo4j

优点：

- 结构更清晰
- 更符合现有 pipeline 风格

建议：

> 如果准备长期保留“子领域层级”这个能力，建议使用 `SubdomainEvolutionAgent`。

## 10. 关系抽取与写库如何统一

这是和导师“仔细看一下他是怎么抽关系的”最相关的部分。

### 10.1 当前问题

现在：

- `Extraction` 输出英文 biomedical relations
- `Neo4jStore._determine_claim_type()` 仍偏向旧中文关系分类

这会导致：

- ontology relation 可能不能被正确识别
- hierarchy/type/property 写入分类不稳定

### 10.2 推荐改法

统一采用英文关系 schema。

建议在 `Neo4jStore` 中把 ontology relation 分类改成基于以下英文关系：

#### ontology_subclass

- `subclass_of`

#### ontology_type

- `type`

#### ontology_property

- `biomarker_for`
- `risk_factor_for`
- `part_of`
- `associated_with`
- `measured_in`
- `inhibits`
- `activates`
- `treats`

#### instance

- `has_diagnosis`
- `received_treatment`
- `combined_with`
- `measured_value`
- `has_adverse_event`
- `predicts`
- 等

### 10.3 子领域关系也统一到同一体系

建议子领域层级关系也统一使用：

- `SUBCLASS_OF`

而不要在 Neo4j 里继续保留旧中文关系类型：

- `子类`
- `类型`

否则：

- biomedical claim 关系是一套
- subdomain hierarchy 又是一套
- 后期会越来越乱

---

## 11. 推荐实施顺序

建议按下面顺序落地，而不是一次性全改。

### 第一阶段：把子领域正式写进图

目标：

- `Paper` 节点有 `subdomain` 字段
- `Paper -[:CLASSIFIED_AS]-> Subdomain`
- `Subdomain -[:SUBCLASS_OF]-> Parent`

这是当前最先要满足导师要求的最小闭环。

### 第二阶段：把子领域 hierarchy 的来源改成 Neo4j

目标：

- 分类前从图里读 hierarchy
- 分类后把新层级写回图
- 减少静态 YAML 的作用

### 第三阶段：统一 relation schema

目标：

- `Extraction` 关系
- `Neo4j` 存储关系
- `Validation` 判断关系

三者统一为同一套英文 biomedical schema

### 第四阶段：再优化分类与抽取效果

目标：

- 降低子领域“写死感”
- 降低 Extraction 的过抽取

---

## 12. 风险与注意事项

### 风险 1：子领域无限生长

如果完全放开让模型新建子领域，可能出现：

- 粒度过细
- 同义子领域过多
- hierarchy 很快膨胀

建议：

- 新增子领域前，先和现有节点做相似性匹配
- 必要时加一层人工审核或 merge 规则

### 风险 2：parent_domain 挂接不稳定

同一子领域可能被不同轮次挂到不同父领域。

建议：

- 对已有子领域优先复用
- 对新子领域的 parent 选择加 stricter prompt
- 在图中记录 `source` 和 `confidence`

### 风险 3：relation schema 继续漂移

如果不先统一抽取关系和写库关系，后面图谱会越来越难维护。

所以：

> 子领域入图和 relation schema 统一，最好并行考虑。

---

## 13. 最终建议

一句话总结：

> 当前系统已经有“子领域标签”的基础，但还没有真正实现“子领域入图 + 层级维护 + 图谱驱动分类”。下一步最合理的改法，是把子领域从 `metadata` 升级为 Neo4j 中的正式层级对象，同时把当前静态配置降级为兜底参数，把 hierarchy 的主要来源改成图数据库本身。

最优先建议：

1. 先把 `Paper / Subdomain / CLASSIFIED_AS / SUBCLASS_OF` 写进 Neo4j
2. 再让 `SubdomainClassifierAgent` 从 Neo4j 读已有 hierarchy
3. 再统一 `Extraction -> Neo4jStore` 的 relation schema

这样才真正符合导师的三句话：

- 在 neo4j 里加上子领域字段
- 并维护子领域之间关系
- 且不要把子领域配置写死
